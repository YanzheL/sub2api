# ENT CODEGEN GUIDE

## OVERVIEW
`backend/ent/` is the generated ORM boundary. In normal work, you edit schema definitions under `schema/`, then regenerate the rest.

## STRUCTURE
```text
backend/ent/
├── schema/         # handwritten entity definitions
├── schema/mixins/  # shared schema behavior
├── migrate/        # generated migration schema helpers
├── runtime/        # generated runtime helpers
├── hook/           # generated hooks
├── intercept/      # generated interceptors
└── *.go            # generated client/query/mutation/entity files
```

## WHERE TO LOOK
| Task | Location | Notes |
|------|----------|-------|
| Add/remove fields | `schema/*.go` | Handwritten source of truth |
| Shared schema behavior | `schema/mixins/` | Timestamps, soft-delete-like shared patterns |
| Generated CRUD/query output | top-level generated files | Review only; do not author here |
| SQL evolution | `../migrations/` | SQL migrations remain authoritative runtime history |

## CONVENTIONS
- Assume every non-`schema/` file here is machine-managed unless the file content clearly says otherwise.
- Schema changes are not complete until generated Ent output is refreshed and reviewed.
- SQL migrations and Ent schema are related but not interchangeable: Ent defines model shape; `backend/migrations/` defines forward runtime history.

## COMMANDS
```bash
cd backend && go generate ./ent
cd backend && go generate ./cmd/server
```

## ANTI-PATTERNS
- Do not hand-edit files marked `Code generated by ent, DO NOT EDIT.`
- Do not treat `migrate/` output as handwritten migration logic.
- Do not change schema files without regenerating and reviewing the downstream diff.
- Do not “fix” an applied migration by editing old SQL; add a new migration file.

## REVIEW CHECKLIST
- Did the actual source change happen in `schema/`?
- Was `go generate ./ent` run from `backend/`?
- Did the generated diff stay within expected entities/edges/indexes?
- Does a runtime/data migration also need a new file under `backend/migrations/`?
