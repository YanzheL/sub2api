# BACKEND GUIDE

## OVERVIEW
`backend/` contains the Go service, handwritten application layers under `internal/`, Ent schema/codegen, and forward-only SQL migrations.

## STRUCTURE
```text
backend/
├── cmd/server/        # main(), setup mode, Wire bootstrap
├── ent/               # generated Ent code + handwritten schema/
├── internal/
│   ├── config/        # config loading and validation
│   ├── handler/       # HTTP handlers and DTOs
│   ├── repository/    # DB/cache/external persistence layer
│   ├── server/        # router, routes, server middleware
│   ├── service/       # business logic and gateway behavior
│   ├── setup/         # setup wizard / auto-setup helpers
│   ├── pkg/           # reusable backend packages
│   └── util/          # narrow utility helpers
└── migrations/        # SQL migration files embedded at startup
```

## WHERE TO LOOK
| Task | Location | Notes |
|------|----------|-------|
| Process startup | `cmd/server/main.go` | Flags, setup mode, graceful shutdown |
| DI graph | `cmd/server/wire.go` | Generated output lands in `wire_gen.go` |
| Router setup | `internal/server/router.go` | Middleware + route registration |
| Route groups | `internal/server/routes/` | Common/auth/user/admin/gateway split |
| HTTP handler entry | `internal/handler/` | `admin/` subtree for admin-only endpoints |
| Business logic | `internal/service/` | Largest handwritten hotspot in repo |
| Persistence/caches | `internal/repository/` | Ent, Redis, migration runner, repo helpers |
| Setup flows | `internal/setup/` | CLI/web setup and auto-setup behavior |
| Data model/schema | `ent/schema/` and `migrations/` | Schema + SQL evolution live separately |

## CONVENTIONS
- Treat `handler -> service -> persistence` as the practical flow, even though lint rules forbid `internal/service` and `internal/handler` from directly importing `internal/repository`.
- Keep direct persistence details in `internal/repository/`; handlers should not grow DB/cache access logic.
- `internal/service/` and `internal/repository/` are both intentionally flat and large. Prefer extending an existing file in the relevant domain before creating new micro-packages.
- Test naming is meaningful here: `_test.go` is common, and some files use `_integration_test.go` / `_unit_test.go` to signal scope.
- `backend/.golangci.yml` enforces architectural boundaries; if a change feels like it crosses layers, assume the lint config will object.
- Migrations are forward-only and effectively ordered by filename; preserve existing naming patterns like `042b_*` and treat `*_notx.sql` files as special non-transactional cases.

## GENERATED BOUNDARIES
- See `backend/ent/AGENTS.md` before touching anything under `ent/`.
- `cmd/server/wire_gen.go` is generated by Wire from `cmd/server/wire.go` and provider sets.
- `migrations/` is embedded and checksum-tracked at runtime; applied SQL is immutable.

## COMMANDS
```bash
cd backend && make generate
cd backend && make build
cd backend && make test
cd backend && make test-unit
cd backend && make test-integration
cd backend && make test-e2e
```

## ANTI-PATTERNS
- Do not hand-edit `wire_gen.go`.
- Do not bypass repository-layer persistence by scattering raw DB/Redis logic through handlers.
- Do not rewrite applied migration files to fix production data/model issues; add a new migration.
- Do not edit generated Ent outputs when the real source of truth is `ent/schema/`.

## NOTES
- `internal/service/gateway_service.go`, `openai_gateway_service.go`, and related gateway files are the core backend hotspots.
- `internal/repository/usage_log_repo.go` is the main persistence hotspot for usage/billing data.
- The setup path is first-class: `main.go` can run CLI setup, web setup, or auto-setup depending on flags and environment.
- If you need a backend binary that serves the frontend directly, a normal `make build` is not enough; use an embed-capable build path.
